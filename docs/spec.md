# Техническое задание: улучшение изображений в браузере

## 1. Цель

Разработать систему улучшения изображений, работающую целиком в браузере
пользователя: ML-модель подбирает оптимальные параметры коррекции яркости,
контрастности и цветности по уменьшенной копии изображения, а быстрый
вспомогательный алгоритм применяет эти параметры к полноразмерному изображению.

## 2. Требования (из постановки задачи)

| № | Требование | Как выполняется |
|---|---|---|
| 1 | Работоспособность во всех массовых современных браузерах | Целевые движки: WebKit (Safari), Chromium (Chrome, Яндекс Браузер), Gecko (Zen/Firefox). Только стабильные API: Web Worker, OffscreenCanvas 2d, createImageBitmap. Без WebGPU/SharedArrayBuffer |
| 2 | Суммарный объём кода ≤ 10 МБ | Собственный инференс-движок (~10 КБ) вместо TF.js; веса fp16 ~400 КБ; libheif загружается лениво (~2 МБ). Итого ≈ 2.6 МБ; CI-гейт на 8 МБ |
| 3 | Изображения до 15 Мпк | Тайловая обработка in-place, один буфер 60 МБ, transferable-передача |
| 4 | Макс. время обработки ≤ 30 с | Watchdog на главном потоке: по истечении 30 с задача завершается с ошибкой, worker пересоздаётся |
| 5 | Среднее время ≈ 5 с | Один проход LUT+люма по 15 Мпк ≈ 0.3–1 с; декодирование/кодирование — нативные API |
| 6 | Форматы JPG, PNG, HEIC, BMP | JPG/PNG — createImageBitmap; BMP — нативно + собственный запасной декодер; HEIC — нативно в Safari, иначе libheif (WASM, лениво) |
| 7 | Асинхронный режим | Вся обработка в выделенном Web Worker; главный поток только диспетчеризует события |

## 3. Архитектура

```
Главный поток                       Worker (постоянный, один)
┌───────────────────────┐           ┌─────────────────────────────────┐
│ ImageEnhancer (API)   │  submit   │ 1. Определение формата (magic)  │
│  - реестр задач       │ ────────► │ 2. Декодирование → RGBA         │
│  - FIFO-очередь       │           │ 3. Миниатюра 224×224            │
│  - события            │  status/  │ 4. CNN → (β, γ, σ)              │
│  - watchdog 30 с      │ ◄──────── │ 5. Тайловое применение (LUT)    │
│  - отмена             │  progress │ 6. Кодирование → Blob           │
└───────────────────────┘  /result  └─────────────────────────────────┘
```

- Одна задача в обработке одновременно (предсказуемая память: один буфер 15 Мпк).
- Отмена: флаг проверяется в точках yield между тайлами и стадиями
  (задержка отмены ≤ ~100 мс вне нативных вызовов декодера/кодера).
- Буферы передаются между потоками как transferable (без копий).

## 4. Программный интерфейс (API модуля)

```ts
class ImageEnhancer extends EventTarget {
  /** Постановка задачи. Возвращает идентификатор немедленно. */
  submitTask(image: File | Blob | ArrayBuffer): string;
  /** Текущий статус и прогресс (0–100). */
  getStatus(taskId: string): { status: TaskStatus; progress: number; error?: string };
  /** Прерывание задачи. */
  cancelTask(taskId: string): { success: boolean; reason?: string };
  /** Готовое изображение (бросает исключение, если статус не 'done'). */
  getResult(taskId: string): Blob;
}

type TaskStatus = 'queued' | 'decoding' | 'analyzing' | 'enhancing'
                | 'encoding' | 'done' | 'error' | 'cancelled';

// Событие изменения статуса/прогресса:
enhancer.addEventListener('taskstatuschange',
  (e: CustomEvent<{ taskId: string; status: TaskStatus; progress: number }>) => {});
```

Шкала прогресса: queued 0 · decoding 0→20 · analyzing 20→35 ·
enhancing 35→90 (по тайлам) · encoding 90→99 · done 100.

Формат результата: JPEG (q≈0.92) для входных JPG/HEIC; PNG для входных PNG/BMP.

## 5. ML-модель

- CNN, вход 224×224×3 (билинейное сжатие без сохранения пропорций), NHWC:
  5 × [conv 3×3, stride 2, ReLU] (16/32/64/96/128 каналов) → GAP → FC64 ReLU →
  FC3 tanh ≈ 198 тыс. параметров ≈ 400 КБ в fp16.
- Выход `o ∈ [−1,1]³` отображается в (β, γ, σ) — см. `docs/color-math.md`.
- Обучение: самообучение на синтетической деградации (DIV2K): случайные
  тройки коэффициентов деградации (лог-равномерно: β_d∈[0.45,2], γ_d∈[0.5,1.8],
  σ_d∈[0.3,1.8]), ~10% полностью неискажённых примеров, шум σ≤0.01,
  JPEG-пережатие q70–95. Лосс: L1 в пространстве изображения
  (apply(degraded, predicted) против оригинала, мягкий clamp для градиентов)
  + 0.1·MSE по параметрам с затуханием до 0.
- Инференс в браузере: собственный движок (conv2d/relu/gap/dense/tanh,
  Float32Array, NHWC), валидируется послойными golden-векторами из Keras.
- Запасной вариант: модель обучена в Keras ⇒ `tensorflowjs_converter` + TF.js
  без переобучения.

## 6. Критерии приёмки

1. Все golden-тесты цветовой математики (±1 LSB) и NN (≤1e-4) проходят.
2. E2E: JPG/PNG/BMP/HEIC обрабатываются во всех трёх движках (Playwright:
   chromium/webkit/firefox); прогресс монотонный; отмена в середине
   обработки даёт статус `cancelled`; 15 Мпк < 30 с.
3. Среднее время по эталонному пулу ≤ 5 с (замер `?bench`).
4. Объём опубликованного решения ≤ 10 МБ (CI-гейт 8 МБ).
5. ML-режим не ухудшает уже качественные изображения
   (метрика identity safety в `evaluate.py`).
6. Качество: ML-режим превосходит эвристический базовый алгоритм по
   PSNR/SSIM на синтетических деградациях эталонного пула.

## 7. Эталонный пул изображений

`reference-images/` + `manifest.md`: недо-/переэкспонированные, малоконтрастные,
обесцвеченные, перенасыщенные, ночные, портрет, 3 эталонно-хороших;
стресс-кейсы: 15 Мпк JPEG и PNG, HEIC, BMP. Источники и лицензии — в манифесте.

## 8. Эквивалентность браузеров

Zen построен на Gecko (Firefox ESR), Яндекс Браузер — на Chromium.
Автоматические тесты на движках playwright-webkit/chromium/firefox покрывают
все четыре целевых браузера; дополнительно выполняется ручная проверка в
реальных Safari, Zen и Яндекс Браузере (фиксируется в `docs/decisions.md`).
